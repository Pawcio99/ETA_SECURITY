Abstract / Streszczenie

Wielowarstwowa izolacja sieciowa punktów końcowych

Abstract (EN)

This paper presents the Enforced Tunnel Architecture (ETA) — a multi-layer network security system designed for endpoint devices operating in untrusted network environments. ETA integrates WireGuard VPN with a fail-closed nftables kill-switch, a cryptographically validated DNS stack (Unbound + DNSSEC + dnscrypt-proxy/DoH), kernel-level IPv6 suppression, and optional application isolation via QEMU/KVM. The complete system was implemented and verified on Kali Linux 2024.x. An automated 88-test security audit yielded a score of 86/88 PASS (grade A — Excellent) with zero critical failures and zero FAIL results. One WARN pertains to browser applications running on the host rather than inside the isolated VM. Results confirm that ETA eliminates all tested attack vectors: DNS leakage, VPN bypass, IPv6 tunnel bypass, and passive traffic interception. The paper additionally presents ETA-Stealth — an extended configuration integrating ProtonVPN Stealth protocol (TLS obfuscation over TCP 443) that conceals the VPN node from Deep Packet Inspection (DPI) and network-level censorship. The architecture is fully open-source and reproducible.

Streszczenie (PL)

Artykuł opisuje Enforced Tunnel Architecture (ETA) — wielowarstwowy system bezpieczeństwa sieci dla urządzeń końcowych pracujących w niezaufanych środowiskach sieciowych. ETA integruje VPN WireGuard z kill-switchem nftables działającym według zasady fail-closed, kryptograficznie uwierzytelnionym stosem DNS (Unbound + DNSSEC + dnscrypt-proxy/DoH), wyłączeniem IPv6 na poziomie jądra oraz opcjonalną izolacją aplikacji przez QEMU/KVM. System zaimplementowano i zweryfikowano na Kali Linux 2024.x. Automatyczny audyt 88 testów bezpieczeństwa dał wynik 86/88 PASS (ocena A — Doskonały) przy zerowej liczbie błędów krytycznych i zerowej liczbie wyników FAIL. Artykuł prezentuje ponadto rozszerzenie ETA-Stealth: integrację protokołu Stealth ProtonVPN (obfuskacja TLS przez TCP 443), która ukrywa węzeł VPN przed głęboką inspekcją pakietów (DPI) i cenzurą sieciową.

Section 01

Wprowadzenie

Współczesne środowiska pracy — zdalna, hybrydowa, mobilna — wymagają od urządzeń końcowych łączności z zasobami firmowymi za pośrednictwem sieci, nad którymi organizacja nie sprawuje żadnej kontroli. Hotele, lotniska, kawiarnie i sieci klientów stanowią środowisko aktywnie wrogie dla niezabezpieczonych połączeń.

Tradycyjne rozwiązania ochrony punktów końcowych — oprogramowanie antywirusowe, zapora systemowa, protokół HTTPS — adresują wyłącznie zagrożenia zlokalizowane na urządzeniu lub w warstwie aplikacyjnej. Nie chronią one warstwy sieciowej, przez którą dane fizycznie przepływają między urządzeniem a siecią zewnętrzną.

Section 02

Metody i implementacja

2.1 Architektura warstwowa ETA

ETA definiuje pięć niezależnych warstw ochrony, z których każda eliminuje odrębną klasę zagrożeń. Warstwy są wzajemnie wzmacniające — kompromitacja jednej z nich nie powoduje naruszenia pozostałych.

2.2 Implementacja stosu DNS

Stos DNS składa się z trzech komponentów działających jako łańcuch kryptograficznego uwierzytelnienia. Systemowa usługa systemd-resolved jest zatrzymana i zamaskowana. Plik /etc/resolv.conf jest zastępowany statycznym wpisem wskazującym na 127.0.0.1. 

# /etc/resolv.conf — statyczny, chroniony przez tmpfiles.d
nameserver 127.0.0.1
options edns0 trust-ad

# Unbound nasłuchuje na 127.0.0.1:53 i 10.0.3.1:53
# Zapytania upstream → dnscrypt-proxy na 127.0.0.1:5053
# dnscrypt-proxy → DoH przez wg0 (szyfrowany tunel)

# Test walidacji DNSSEC (musi zwrócić SERVFAIL):
$ dig @127.0.0.1 dnssec-failed.org A
;; ->>HEADER<<- opcode: QUERY, status: SERVFAIL

Kluczowe parametry konfiguracji Unbound: val-permissive-mode: no (twarda walidacja DNSSEC — brak odpowiedzi zamiast sfałszowanego wyniku), qname-minimisation: yes (RFC 7816) oraz do-not-query-localhost: no (zezwolenie na zapytania do dnscrypt-proxy na loopback).

2.3 WireGuard i kill-switch nftables

Kill-switch jest ładowany jako usługa systemd z dyrektywą Before=network-pre.target, co gwarantuje aktywność reguł firewall przed konfiguracją interfejsów sieciowych przez kernel. Polityka domyślna OUTPUT: drop blokuje każdy pakiet wychodzący, z wyjątkiem: 

table inet filter {
  chain output {
    type filter hook output priority 0; policy drop;

    # loopback — komunikacja między usługami lokalnymi
    oifname "lo"   accept

    # jedyny dozwolony ruch przez interfejs fizyczny:
    # handshake WireGuard (UDP → IP endpointu VPN)
    oifname "eth0" ip daddr $VPN_ENDPOINT udp dport $WG_PORT accept

    # ruch przez tunel VPN
    oifname "wg0"  accept

    # ruch z/do maszyny wirtualnej przez TAP
    oifname "tap0" accept
    iifname "tap0" oifname "wg0" accept
  }
}

2.4 Izolacja aplikacji w QEMU/KVM

Opcjonalna warstwa izolacji uruchamia przeglądarkę LibreWolf w maszynie wirtualnej Debian 12 z dostępem do sieci wyłącznie przez interfejs TAP połączony z hostem. Maszyna wirtualna nie ma dostępu do klucza WireGuard, konfiguracji hosta ani lokalnego systemu plików. Kompromitacja przeglądarki jest ograniczona do środowiska VM.

✓

Dyskusja

4.1 Zasada fail-closed jako fundamentalna gwarancja

Kluczową innowacją architektury ETA jest konsekwentna implementacja zasady fail-closed na każdym poziomie stosu. Tradycyjne rozwiązania VPN domyślnie działają w trybie fail-open — utrata połączenia powoduje automatyczne przełączenie na połączenie bezpośrednie. W ETA utrata połączenia oznacza brak internetu, nie wyciek.

4.3 ETA-Stealth — ukrycie węzła VPN przed DPI

Standardowy ruch WireGuard jest rozpoznawalny przez techniki głębokiej inspekcji pakietów (DPI — Deep Packet Inspection). Charakterystyczny nagłówek UDP protokołu WireGuard oraz stały port (domyślnie 51820/UDP) umożliwiają klasyfikację i blokowanie ruchu przez firewalle korporacyjne, systemy IDS/IPS, a także przez mechanizmy cenzury sieciowej stosowane na poziomie ISP lub krajowej infrastruktury sieciowej.

ProtonVPN Stealth to protokół obfuskacji, który opakowuje ruch WireGuard w strumień TLS przez TCP 443. Z perspektywy każdego urządzenia monitorującego sieć połączenie wygląda identycznie jak zwykłe połączenie HTTPS do dowolnej strony internetowej. Nie ma możliwości odróżnienia go od normalnego ruchu webowego bez klucza prywatnego.

┌──────────────────────────────────────────────────────────────────┐ │ ETA-STEALTH — przepływ ruchu │ └──────────────────────────────────────────────────────────────────┘ Aplikacja (VM / host) │ dowolny protokół (TCP/UDP) ▼ WireGuard (wg0) ←— szyfrowanie ChaCha20-Poly1305, tunelowanie │ UDP wewnątrz TLS ▼ ProtonVPN Stealth Transport Layer │ obfuskacja: WireGuard UDP → strumień TLS │ port: TCP 443 (nieodróżnialny od HTTPS) ▼ Sieć (hotel / lotnisko / ISP / firewall korporacyjny / DPI) │ "widzę: HTTPS do IP X" — brak możliwości identyfikacji VPN ▼ Serwer ProtonVPN (węzeł Stealth) │ deszyfracja TLS → deszyfracja WireGuard ▼ Internet ──── Co widzi obserwator sieci ───────────────────────────────────── Standardowy WireGuard: UDP:51820 → IP serwera ProtonVPN [ROZPOZNAWALNY] ETA-Stealth: TCP:443 → IP serwera (brak podpisu VPN) [NIEWIDOCZNY]

4.3.1 Konfiguracja ProtonVPN Stealth na Kali Linux

ProtonVPN Stealth jest dostępny w planach Plus i Visionary. Wymaga użycia oficjalnego klienta ProtonVPN dla Linux lub ręcznej konfiguracji przez dedykowany plik .ovpn / .conf w trybie Stealth. Poniżej przedstawiono konfigurację integrującą Stealth z architekturą ETA: 

# Krok 1 — Pobierz konfigurację Stealth z protonvpn.com
# Panel → Downloads → Linux → WireGuard → Stealth protocol
# Lub: protonvpn-cli c --stealth (oficjalny klient)

# Krok 2 — Instalacja klienta ProtonVPN CLI (Debian/Kali)
wget https://repo.protonvpn.com/debian/dists/stable/main/binary-all/protonvpn-stable-release_1.0.3-3_all.deb
sudo dpkg -i protonvpn-stable-release_1.0.3-3_all.deb
sudo apt update && sudo apt install proton-vpn-gnome-desktop

# Lub CLI bez GUI:
sudo apt install protonvpn-cli

# Krok 3 — Połączenie w trybie Stealth przez CLI
protonvpn-cli login --username USER
protonvpn-cli connect --stealth --fastest

# Krok 4 — Integracja z ETA kill-switch
# W pliku /etc/nftables-vpn-killswitch.conf zmień:
#   udp dport 51820  →  tcp dport 443
# i zaktualizuj IP endpointu Stealth z pliku konfiguracyjnego

# Weryfikacja — brak rozpoznawalnego podpisu WireGuard:
sudo tcpdump -i eth0 -n 'tcp port 443' | head -5
# Powinno wyglądać jak standardowy ruch TLS
ℹ

4.3.2 Kiedy używać ETA-Stealth

Tryb Stealth nie jest domyślnie zalecany dla wszystkich środowisk — protokół UDP WireGuard zapewnia lepszą wydajność i mniejsze opóźnienia, co ma znaczenie przy pracy z aplikacjami czasu rzeczywistego (wideokonferencje, SSH). ETA-Stealth jest wskazany w następujących scenariuszach:

Scenariusz 1
Sieci korporacyjne z restrykcyjnym firewallem
Środowiska enterprise często blokują ruch UDP inny niż DNS/QUIC oraz wszystkie niestandardowe porty. Stealth (TCP 443) przechodzi przez każdy firewall przepuszczający HTTPS.
Scenariusz 2
Praca w krajach z cenzurą internetową
Systemy cenzury sieciowej (GFW, SORM) identyfikują i blokują ruch VPN na podstawie analizy protokołów. Obfuskacja TLS sprawia, że VPN jest nieodróżnialny od ruchu webowego.
Scenariusz 3
Dziennikarstwo, działalność śledcza, whistleblowing
Gdy sama obecność połączenia VPN w logach ISP może stanowić dowód działalności, ukrycie węzła VPN eliminuje ten ślad metadanych.
Scenariusz 4
Sieci z aktywnym monitoringiem (Red Team / Pen Test)
W środowiskach z zaawansowanym SOC i SIEM, gdzie anomalie sieciowe wyzwalają alerty, Stealth eliminuje sygnaturę VPN z telemetrii sieciowej.

4.4 Ograniczenia i znane luki

Architektura ETA (w tym wariant Stealth) nie eliminuje wszystkich możliwych wektorów ataku. Zidentyfikowane ograniczenia: (1) brak ochrony przed atakami na poziomie urządzenia (keylogger, rootkit na hoście — mitigacja: Etap 3, pełna izolacja VM); (2) zależność od infrastruktury dostawcy VPN — kompromitacja serwerów ProtonVPN byłaby poza kontrolą ETA; (3) Stealth eliminuje identyfikację VPN, ale nie ukrywa faktu połączenia z konkretnym adresem IP serwera ProtonVPN (mitigacja: zmiana serwera, routing przez Tor); (4) brak ochrony metadanych czasowych i wolumenu transmisji.

⚠

Ważne zastrzeżenie: ETA jest narzędziem ochrony prywatności i bezpieczeństwa sieci dla legalnych zastosowań. System testowano wyłącznie w środowisku własnym autora. Wszelkie testy penetracyjne należy przeprowadzać wyłącznie na systemach, do których posiada się autoryzację.

Section 05

Wnioski

Badanie potwierdza hipotezę, że możliwe jest zbudowanie wielowarstwowej architektury bezpieczeństwa sieci wyłącznie z komponentów open-source, która eliminuje wszystkie zidentyfikowane wektory wycieku danych w środowisku niezaufanej sieci zewnętrznej.

Wynik 86/88 PASS przy ocenie A — Doskonały, zerowej liczbie błędów krytycznych i zerowej liczbie wyników FAIL potwierdza dojrzałość implementacji do zastosowań produkcyjnych. Jedyny WARN (przeglądarka na hoście zamiast w VM) znika po aktywacji Etapu 3. Architektura ETA jest gotowa do wdrożenia produkcyjnego.

Rozszerzenie ETA-Stealth z protokołem ProtonVPN Stealth (obfuskacja TLS przez TCP 443) eliminuje ostatni identyfikowalny element architektury — rozpoznawalną sygnaturę UDP WireGuard. Po aktywacji Stealth żaden obserwator sieci, DPI ani system cenzury nie jest w stanie odróżnić połączenia ETA od standardowego ruchu HTTPS, co czyni architekturę kompletną dla najbardziej wymagających scenariuszy operacyjnych.

Kluczowe wnioski techniczne: (1) zasada fail-closed wymaga załadowania kill-switcha jako usługi systemd z Before=network-pre.target; (2) wyłączenie systemd-resolved i statyczny /etc/resolv.conf są warunkiem koniecznym ochrony DNS; (3) linia DNS= w pliku WireGuard musi być zakomentowana; (4) przy przejściu na Stealth reguła nftables wymaga zmiany udp dport 51820 → tcp dport 443 z aktualizacją IP endpointu.

Szybkie wdrożenie (Kali Linux / Debian)

# Sklonuj repozytorium
git clone https://github.com/majsterek/eta.git
cd eta

# Etap 1 + 2: Host hardening, DNS, VPN, kill-switch (~45 min)
sudo bash eta_install.sh --vpn-config /path/to/wg0.conf

# Weryfikacja bezpieczeństwa (88 testów)
sudo bash eta_security_test.sh

# Opcjonalnie: przełączenie na ETA-Stealth (ProtonVPN Stealth / TCP 443)
sudo bash eta_stealth_switch.sh --stealth-config /path/to/wg0-stealth.conf

# Opcjonalnie: izolacja VM (Etap 3)
sudo bash eta_vm_setup.sh --image debian12-cloud.qcow2
